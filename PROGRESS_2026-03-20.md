## 2. Data

### Source
Yahoo Finance via `yfinance` library.

### Universe
3 US equities / indices:
- **AAPL** (Apple Inc.) — large-cap tech stock
- **MSFT** (Microsoft Corp.) — large-cap tech stock
- **SPY** (S&P 500 ETF) — broad market index

### Date Range
- **Raw data:** 2020-01-01 → 2024-01-01
- **After feature warmup (50-day SMA):** 2020-03-13 → 2023-12-29
- **Rows per ticker:** 957 trading days
- **Total pooled rows:** 2,871

### Train / Test Split
- **Method:** Time-based (strict chronological, no shuffling)
- **Train:** 2020-03-13 → 2023-03-27 (765 rows per ticker, 2,295 pooled)
- **Gap:** 30 calendar days (simulates deployment delay)
- **Test:** 2023-04-27 → 2023-12-29 (171 rows per ticker, 513 pooled)
- **Test period:** ~9 months of out-of-sample data

### Data Format
OHLCV (Open, High, Low, Close, Volume) daily bars.
Auto-adjusted for splits and dividends.
Stored locally as CSV in `/data/`.

## 3. Features

All features are **strictly point-in-time** — no lookahead bias.
Computed independently per ticker before pooling.

| Feature | Formula | What It Captures |
|---|---|---|
| `return_1d` | `close.pct_change(1)` | Yesterday's momentum |
| `return_5d` | `close.pct_change(5)` | Weekly momentum |
| `return_10d` | `close.pct_change(10)` | Bi-weekly momentum |
| `return_21d` | `close.pct_change(21)` | Monthly momentum |
| `log_return_1d` | `log(close / close[t-1])` | Daily return (normally distributed) |
| `volatility_5d` | `rolling(5).std(log_return)` | Short-term risk regime |
| `volatility_21d` | `rolling(21).std(log_return)` | Medium-term risk regime |
| `price_to_sma10` | `close / SMA(10)` | Short-term trend position |
| `price_to_sma50` | `close / SMA(50)` | Medium-term trend position |
| `sma10_to_sma50` | `SMA(10) / SMA(50)` | Golden/death cross signal |
| `rsi_14` | Wilder RSI, 14-day EWM | Overbought/oversold |
| `volume_change_1d` | `volume.pct_change(1)` | Volume surge detection |
| `volume_ratio` | `volume / SMA_volume(10)` | Unusual volume activity |

**Total: 13 features**

### Target Variable
```
target = 1 if close[t+1] > close[t] else 0
```
Binary classification: predict whether tomorrow's close is higher than today's.
The shift is applied **after** all features are computed to prevent leakage.

## 4. Models

### Model 1: Logistic Regression (Baseline)
- **Pipeline:** `StandardScaler → LogisticRegression`
- **Why scaled:** LR uses gradient descent — unscaled features produce poor weights
- **`class_weight="balanced"`:** Corrects for the ~53% up-day bias
- **`max_iter=1000`:** Prevents convergence failures on financial data
- **Probability calibration:** Naturally well-calibrated (no additional step needed)

### Model 2: XGBoost (Calibrated)
- **Base:** `XGBClassifier` with `max_depth=4`, `n_estimators=300`, `learning_rate=0.05`
- **Calibration:** `CalibratedClassifierCV(method="isotonic", cv=5)`
- **Why calibrated:** XGBoost optimises for ranking, not probability accuracy.
  Raw probabilities were unreliable — raising the threshold made results *worse*.
  After isotonic calibration, probabilities reflect true frequencies.
- **Why isotonic not sigmoid:** Financial data is not symmetric.
  Isotonic regression is non-parametric and makes no shape assumptions.

## 6. Key Findings

### Finding 1: Features are the bottleneck
All 13 features are derived from price and volume alone. These are among the most heavily traded signals in existence — they have been largely arbitraged away on daily data by institutional participants. The models are finding weak signal at best.

### Finding 2: Logistic Regression outperforms XGBoost on this data
LR produces consistent, positive Sharpe ratios across all three tickers. XGBoost, even after calibration, makes near-zero trades because it cannot find confident signals. This suggests the relationships in the data are largely linear — there are no strong non-linear interactions for XGBoost to exploit with 13 features and 2,295 training rows.

### Finding 3: XGBoost calibration revealed a critical insight
Before calibration, XGBoost appeared to work (Sharpe 1.618 on SPY) but was making 22 trades using poorly calibrated probabilities. After calibration, it made 0 trades — meaning its true confidence on almost every day was below 50%. The pre-calibration result was an artefact of uncalibrated confidence, not real signal.

### Finding 4: The threshold effect is asset-specific
- AAPL responds well to threshold 0.55 for LR (Sharpe jumps from -0.201 to 0.758)
- MSFT and SPY perform best at 0.50 — LR's probability distribution is compressed on these assets and rarely exceeds 0.55

### Finding 5: Risk-adjusted results are more promising than raw return
The strategies never beat buy-and-hold on raw return, but they do so with meaningfully lower drawdowns. LR on MSFT captures 12.24% vs 24.14% buy-and-hold, but with only -8.54% max drawdown vs MSFT's significantly larger swings in 2023.

### Finding 6: Win rate alone is meaningless
XGBoost on MSFT had 61.5% win rate but lost 22% overall (before calibration). The win rate trap — winning small, losing large — is a real risk with uncalibrated tree models.

### Finding 7: SPY is more predictable than individual stocks
All best Sharpe ratios are on SPY or closely contested. Index-level signals from price/volume features are more stable than stock-specific ones. Individual stocks have more event-driven noise (earnings, news) that these features cannot capture.

---

## 7. What Does NOT Work Yet

- **XGBoost with these features:** Post-calibration makes near-zero trades. Needs richer features to form confident views
- **Single-ticker specialisation:** The pooled model treats AAPL and SPY identically — they have very different price dynamics
- **Raw return vs buy-and-hold:** Strategies consistently underperform on total return — only competitive on risk-adjusted basis
- **9 months test data:** Statistically insufficient to distinguish signal from noise with confidence. Need walk-forward validation with multiple out-of-sample periods to make robust claims

## 9. Planned Improvements — In Priority Order

### Priority 1: Richer Features
The single highest-value improvement. Current features are all derived from price/volume. Need to add:

**a) Market regime feature**
- SPY 21-day rolling volatility as a market-wide signal
- VIX level (fear index) — tells the model if the market is stressed
- This single feature could meaningfully improve all three tickers

**b) Cross-sectional momentum**
- Rank each ticker's return relative to the others
- "AAPL is in the top 30% of performers this week" is more signal than "AAPL is up 2%"

**c) Longer lookback returns**
- 63-day (quarterly) and 126-day (semi-annual) momentum
- Academic literature shows these have more persistence than short-term returns

**d) Earnings proximity** *(stocks only)*
- Days to next earnings announcement
- Models should treat pre/post-earnings periods differently

### Priority 2: Per-Asset Threshold Optimisation
Use the training set to find the optimal threshold per ticker/model combination.
Not overfitting — we only use training data to select it.
Expected impact: meaningful improvement on AAPL specifically.

### Priority 3: Walk-Forward Validation
Instead of one 9-month test period, roll the training window forward monthly.
Produces 10-20 out-of-sample periods instead of 1.
Makes all Sharpe ratio and performance claims statistically meaningful.
**This is the step that separates proof-of-concept from trusted research.**

### Priority 4: Per-Ticker Models
Train AAPL-specific, MSFT-specific, SPY-specific models instead of one pooled model.
Each model learns the characteristics of that asset.
Expected to help most on AAPL where the pooled model performs worst.

### Priority 5: Probability Calibration Verification
Add a calibration curve plot (reliability diagram) to the evaluator.
Visually verify that P(up)=0.6 actually corresponds to ~60% up days.
Currently we apply calibration but don't verify it worked.

### Priority 6: Portfolio Construction
Trade all three tickers simultaneously.
Size positions by model confidence (higher confidence = larger position).
Add a portfolio-level Sharpe and drawdown metric.
